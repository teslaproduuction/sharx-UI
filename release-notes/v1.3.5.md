## Release v1.3.5

**Release Date:** May 11, 2026

Релиз относительно **v1.3.4**: на worker-узлы добавлена интеграция с **Docker-обновлятором (Watchtower)** — кнопка «обновить» в панели теперь дёргает не только панельный sidecar, но и updater на всех включённых worker-узлах; на странице **Nodes** появилась колонка **версии worker-сборки** (`sharxVersion` из `/api/v1/status`, кэшируется в БД). Переписана работа с **WireGuard-клиентами**: для отключённых/просроченных клиентов peer-строки больше не пропадают, а сохраняются в отдельном **panel-only хранилище** `panelWgInactivePeers`, ключи и IP стабильны при повторном включении; Xray-конфиг очищается от этого ключа и `privateKey` через `SanitizeWireGuardSettingsJSONForXray`. **Email клиента** теперь **иммутабелен после создания** (на нём держатся peer-теги WG, статистика, подписки) — поле в UI на редактирование заблокировано, API и `UpdateInboundClient` отвергают смену. Обновлены **README** (EN/RU/FA) и `node/README.md`: pairing/multi-node стали базовым сценарием, упомянут Watchtower-стек на узле и страница подписки. Добавлены утилиты **i18n** для допереводов (`panel/scripts/merge_panel_missing_i18n.py` + JSON-патчи).

Миграции БД: **`0041_node_worker_version.sql`** — колонка **`worker_version TEXT NOT NULL DEFAULT ''`** в **`nodes`** (кешированная сборка worker’а).

### New Features

- **Docker-обновление worker-узлов из панели**
  - На worker: новые endpoints **`GET /api/v1/docker-updater`** (показывает, настроен ли updater) и **`POST /api/v1/docker-updater/trigger`** (вызывает локальный HTTP API сайдкара) (`node/api/server.go`).
  - Логика общая с панелью: вынесена в пакет **`util/dockerupdater`** (`Configured()`, `Trigger(ctx)`); `web/service/docker_updater.go` теперь тонкая обёртка над ней.
  - Панельный контроллер `dockerUpdaterTrigger` после локального вызова идёт по всем включённым узлам через **`NodeService.TriggerDockerUpdaterOnAllNodes(ctx)`**, ошибки логируются и при работающем Telegram-боте уведомляются отдельным сообщением `Node worker docker updater: N error(s)` (`web/controller/server.go`, `web/service/node.go`).
  - Шаблоны `node/docker-compose.yml`, `node/docker-compose.harbor.yml` и compose из модалки добавления узла в UI теперь содержат **watchtower 2.3.2** на bridge-сети `sharx_node_net`, опубликованный на `127.0.0.1:8081`, и переменные **`XUI_DOCKER_UPDATER_URL=http://127.0.0.1:8081/v1/update`** + **`XUI_DOCKER_UPDATER_TOKEN`** (по умолчанию `local-dev-insecure-watchtower-token`, переопределяется `WATCHTOWER_HTTP_API_TOKEN`); worker остаётся в `network_mode: host` и помечен лейблом `com.centurylinklabs.watchtower.enable: "true"` (`node/docker-compose.yml`, `node/docker-compose.harbor.yml`, `panel/components/NodesPage.tsx` — `buildNodeDockerComposeYaml`).

- **Версия worker-сборки на странице Nodes**
  - В `Node` добавлено поле **`WorkerVersion`** (gorm-колонка `worker_version`) (`database/model/model.go`).
  - **`NodeService.SetNodeWorkerVersion`** и `extractNodeWorkerVersion` (читает `sharxVersion`/`workerVersion`) обновляют поле из ответа `/api/v1/status`; при `Enable=false` или ошибке health поле очищается (`web/service/node.go` — `RefreshNodeXrayStateFromWorker`, `CheckNodeHealth`).
  - Worker в **`GET /health`** и **`GET /api/v1/status`** возвращает свою сборку через **`config.GetVersion()`** (`node/api/server.go`).
  - В UI `NodesPage` добавлена колонка **`pages.nodes.workerVersion`** между «Response time» и «Xray version»; ширина таблицы поднята до `min-w-[1040px]`, `colSpan` для «нет совпадений» — 13 (`panel/components/NodesPage.tsx`).

- **WireGuard: «выключенный» клиент не теряет peer**
  - Введён новый ключ настроек инбаунда **`panelWgInactivePeers`** (`PanelWireGuardInactivePeersSettingsKey`) — туда выезжают peer-строки клиентов, которые **назначены** на WG-инбаунд, но сейчас не активны (disabled/expired) (`web/service/wireguard_inbound_settings.go`).
  - **`mergeWireGuardSettingsWithClients`** теперь разделяет `assigned` и `active`: ключи и IP выдаются всем `assigned` (стабильные), а в `settings.peers` остаются только `active`; неактивные клиенты складываются в `panelWgInactivePeers` (`web/service/wireguard_inbound_client_peers.go`).
  - **`SanitizeWireGuardSettingsJSONForXray`** удаляет `panelWgInactivePeers` и `privateKey` из peer’ов перед передачей конфига в Xray, плюс нормализует `address` (`/32`, `/128`) (`web/service/wireguard_inbound_settings.go`).
  - **`PreserveWireGuardPeersOnInboundUpdate`** при пустом `peers` в новом теле также восстанавливает `panelWgInactivePeers` из старых настроек (`web/service/wireguard_inbound_settings.go`).
  - Публичная страница подписки и модалка ключей: помимо активных peer’ов учитывается панель-вальт (`findWireguardPeerForClientActiveOrInactive`), счётчик показывает **`N active + M inactive (panel vault)`** (`sub/wireguard_panel.go`).
  - Покрытие тестами: `web/service/wireguard_inbound_client_peers_test.go`, `web/service/wireguard_inbound_settings_test.go` (sanitize/preserve, merge active/inactive).

- **Email клиента — иммутабелен после создания**
  - В **`web/service/client.go`**: `UpdateClient` отклоняет смену email (`Client email cannot be changed after creation`), нормализованное сравнение через `strings.TrimSpace`/`ToLower`; email в `updates` больше не пишется.
  - В **`web/controller/client.go`** (`updateClient`, оба варианта — JSON и форма): email из тела запроса больше не применяется к существующей сущности.
  - В **`web/service/inbound.go`** (`UpdateInboundClient`): `updatedEntity.Email = clientEntity.Email`, проверка дубликатов email удалена.
  - В UI **`panel/components/ClientsPage.tsx`** поле email в режиме редактирования помечено `readOnly` / `aria-readonly`, с подсказкой `pages.clients.emailImmutableHint` (`Email cannot be changed after the client is created.`).

- **Документация по multi-node и установке**
  - `README.md`, `README_EN.md`, `README_RU.md`, `README_FA.md`: переписан раздел про удалённые узлы — базовый сценарий это **multi-node + pairing**, узлы добавляются через **модалку «добавить узел»** с готовым `docker-compose.yml` (`PANEL_URL` + `SECRET_KEY`), на сервере узла — `docker compose up -d --build`. Параграф про устаревший `NODE_TLS_*` снят как обязательный.
  - Hero-баннер `README.md` и блок Key Features переписаны: вместо «Grafana Integration / Redis Integration» — multi-node, subscription page builder, encrypted cookie sessions, optional observability (Prometheus / Loki / VictoriaMetrics, Grafana dashboard JSON для **вашего** стека).
  - Добавлен раздел **Supported Protocols** (VMess, VLESS, Trojan, Shadowsocks, Hysteria/Hysteria2, Mixed, WireGuard, HTTP/Tunnel, Telemt) и блок про **Subscription Page Builder**.
  - **`node/README.md`** переписан под pairing/multi-node: убран `NODE_API_KEY`, ссылка на `SECRET_KEY` bundle, разделы «Установка узла», «Настройка TLS / сеть», «Переменные окружения», «API (кратко)», «Разработка / сборка образа», структура каталогов и короткая английская инструкция.

- **i18n утилиты для допереводов панели**

  - Базовые патчи: **`panel/scripts/panel_missing_i18n_en.json`**, **`panel/scripts/panel_missing_i18n_ru.json`** (по 256 строк, доперевод недостающих ключей).
  - В результате обновлены сборные **`panel/public/locales/*.json`** (все 12 локалей).

### Operational / Breaking Notes

- **Миграция БД**: после обновления панели применить **`0041_node_worker_version.sql`** — без неё `worker_version` отсутствует и колонка `Worker version` будет пустой.
- **Worker-узлы**: чтобы заработали `/api/v1/docker-updater*` и `sharxVersion` в `/health`/`/api/v1/status`, обновите worker’ы до `1.3.5`. До обновления панель просто не сможет дёргать updater на них и колонка `Worker version` остаётся `—`.
- **Watchtower на узле**: новый шаблон `node/docker-compose.yml` поднимает дополнительный контейнер **`sharx_node_watchtower`** на `127.0.0.1:8081`; в production обязательно задайте **`WATCHTOWER_HTTP_API_TOKEN`** в окружении (значение `local-dev-insecure-watchtower-token` — только для дев-машин). Если на узле уже стоит свой Watchtower / другой updater — оставьте старый compose и просто пропишите `XUI_DOCKER_UPDATER_URL`/`XUI_DOCKER_UPDATER_TOKEN` под него.
- **Email клиента**: интеграции, ранее менявшие `email` через `PUT /panel/api/clients/:id` или через update inbound client, начнут получать ошибку `Client email cannot be changed after creation`. Пересоздавайте клиента, если действительно нужно сменить email.
- **WireGuard**: ключ `panelWgInactivePeers` хранится только в БД (settings JSON инбаунда) и **никогда** не попадает в Xray; ручные правки `settings.peers` через REST/UI совместимы, но не редактируйте `panelWgInactivePeers` вручную — он перегенерируется панелью.
- Если используете JSON ответа `pull-xray-config` / `apply-config`, нужно убедиться что обработчики не падают на дополнительных полях (`sharxVersion`, `workerVersion`); ничего из старых ключей не удалено.
