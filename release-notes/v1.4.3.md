## Release v1.4.3

**Release Date:** May 28, 2026

Релиз **v1.4.3** относительно **v1.4.2**: патч-релиз с тремя исправлениями для multi-node:
(1) ноды теперь идентифицируются панелью по **panel-assigned `nodeId`**, а не по `nodeAddress` — устраняет ситуацию, когда все ноды отображались с `id=1` в логах при общем `SECRET_KEY` и совпадающих API-портах;
(2) приоритет `PANEL_URL` / `NODE_ADDRESS` переменных окружения над персистированным `node-config.json` — устраняет залипание `127.0.0.1:2053` в push-geo после смены `docker-compose`;
(3) **Telemt sidecars** теперь корректно останавливаются на ноде, когда админ снял с неё назначение telemt-инбаунда (раньше старые процессы продолжали работать и блокировали порты, в т.ч. ломали vless на этой ноде).

Миграций БД **нет**.

### Bug Fixes

- **Multi-node: все ноды отображались как `nodeId=1`** (`node/main.go`, `node/api/server.go`, `node/configpull/pull.go`, `node/geopush/geopush.go`, `node/logs/pusher.go`, `node/config/config.go`, `web/service/node.go`, `web/controller/api.go`)
  - **Причина:** панель сопоставляла входящие `push-geo` / `push-logs` / `pull-xray-config` с записью в `nodes` исключительно по `nodeAddress`. При **общем `SECRET_KEY`** и одинаковом API-порте у нескольких воркеров (`127.0.0.1:8080` за разными NAT/доменами) `FindNodeByPushAddress` матчил все запросы в первую запись таблицы — обычно `id=1`.
  - **Фикс:** панель теперь раздаёт нодам **`nodeId` (= `nodes.id` из БД)** в envelope `apply-config` и в ответе `pull-xray-config`. Воркер сохраняет его в `node-config.json` и эхит обратно во **всех** push-/pull-запросах (`pull-xray-config`, `push-logs`, `push-geo`). Хэндлеры на панели сначала ищут по `nodeId`, затем — fallback на `FindNodeByPushAddress` (для первого контакта или legacy-воркеров). Логика `FindNodeByPushAddress` дополнительно ужесточена: точный матч по адресу > порт-only матч **только если уникален**, иначе матча нет.
  - **Эффект:** в логах панели каждая нода теперь имеет свой `nodeId`; счётчики «Last connection node», geo-маркеры, push-logs корректно атрибутируются нужному воркеру даже при общем секрете.

- **Stale `panelUrl` / `nodeAddress` в `node-config.json` побеждали env** (`node/main.go`)
  - **Причина:** старт ноды читал `node-config.json` **до** `os.Getenv` — если файл сохранил `panelUrl: "https://127.0.0.1:2053"` (по умолчанию при первом запуске), новые значения в `docker-compose.yml` игнорировались. В логах появлялось `Geography: push to panel failed: dial tcp 127.0.0.1:2053: connect: connection refused`.
  - **Фикс:** приоритет — `os.Getenv("PANEL_URL")` и `os.Getenv("NODE_ADDRESS")` > `savedConfig.*`. Файл `node-config.json` остаётся источником значений только когда переменная окружения не задана.

- **Telemt sidecars не останавливались после снятия инбаунда с ноды** (`web/service/telemt_config.go`, `web/service/node.go`, `web/controller/api.go`)
  - **Причина:** `BuildTelemtPayloadsForNode` возвращал `nil` (вместо пустого слайса) когда у ноды нет telemt-инбаундов; `ApplyConfigToNode` сериализовал `*telemt` как `null`. Воркер в `apply-config` имеет ветку `if len(telemtRaw) > 0 && string(telemtRaw) != "null"` — то есть `null` трактовался как **«не трогать Telemt»**. Старые telemt-процессы продолжали работать на ноде, занимать порты и ломать **vless** на тех же портах. Ручной `stop-telemt` помогал до следующего `apply-config` / restart, потому что `restart-telemt` оживлял payload из in-memory replay snapshot.
  - **Фикс:** `BuildTelemtPayloadsForNode` возвращает `[]TelemtNodePayload{}` вместо `nil`. `ApplyConfigToNode` форсит непустой пустой слайс при serializе. `pullWorkerXrayConfig` нормализует поле `telemt` до `[]`. Теперь панель отправляет explicit `telemt: []`, что воркер уже корректно обрабатывает как «остановить все Telemt-сайдкары на этой ноде» (см. `Manager.Apply` с `len(payloads) == 0`).
  - **Эффект:** после снятия telemt-инбаунда с ноды и любого `apply-config` / `pull-xray-config` все telemt-процессы на этой ноде останавливаются; порты освобождаются; vless снова работает.

### Improvements

- **API.md:** для `POST /panel/api/node/push-logs` и `POST /panel/api/node/pull-xray-config` задокументированы новые поля `nodeId` (request / response) и уточнено условие на `nodeAddress`.
- **Worker `POST /api/v1/apply-config`:** envelope теперь содержит `nodeId` (панель раздаёт), который воркер обязан персистить в `node-config.json`.

### Operational Notes

- Обновление затрагивает **и панель (`sharx`), и ноды (`sharxnode`)** — обновляйте обе. Сначала панель: новая версия панели обратно совместима со старыми нодами (panel matches по `nodeAddress` если `nodeId` отсутствует). Затем ноды.
- Для существующих воркеров, которые сейчас отображаются как `nodeId=1`, после обновления достаточно одного из:
  - перезапустить ноду (она сделает `pull-xray-config`, получит свой `nodeId` от панели, сохранит в `node-config.json`); или
  - в UI панели нажать «Apply config» для каждой ноды (панель пропатчит `nodeId` через `apply-config`).
- Если на ноде сейчас крутится «лишний» Telemt, который не назначен админом, — обновитесь и сделайте `apply-config` / `pull-xray-config`: теперь telemt останавливается автоматически.
- Если `node-config.json` содержит устаревший `panelUrl` / `nodeAddress` — после обновления значения из env переопределяют файл; ручная очистка `node-config.json` больше не требуется (env сильнее).
- Совместимость HMAC-аутентификации не нарушена; формат подписи и `SECRET_KEY` не менялись.
