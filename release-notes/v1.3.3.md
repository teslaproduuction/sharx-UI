## Release v1.3.3

**Release Date:** May 6, 2026

Релиз относительно **v1.3.2**: публичная страница подписки получила отдельный поток **MTProto (Telegram / `tg://proxy`)** без попадания ссылок в сырой VPN-фид; дашборд и API расширены **историей CPU и памяти по панели и worker-узлам** в multi-node; node-агент отдаёт **хостовые метрики** в статусе; исправлена проверка коллизии портов при назначении инбаундов на узлы; API node/panel переведены на **i18n** для ряда ошибок; масштабные обновления **переводов** (panel locales + backend TOML).

Миграции БД: **нет** (схема не менялась).

### New Features

- **MTProto на публичной подписке (без загрязнения VPN-потока)**
  - Backend: хук `RegisterPublicSubMtProtoHook` и поле **`mtProtoLinks`** в ответе `GET /panel/api/public/subscription` (`web/controller/public_subscription.go`, регистрация в `main.go`).
  - Генерация строк: `SubService.TelemtTgProxyLinesForSubscription` подбирает Telemt-инбаунды клиента и возвращает только строки **`tg://proxy…`** (`sub/subService.go`). Сырое тело подписки (`links` / GetSubs) по-прежнему не содержит MTProto для VPN-клиентов.
  - Типы и UI: `PublicSubPayload.mtProtoLinks`, `resolveMtProtoLinks` / `extractTgProxyLinks` (`panel/components/sub/types.ts`).
  - Каталог приложений подписки: новый пункт **`telegram`** с deeplink по `tg://` (`panel/lib/sharxSubpageConfig.ts`).
  - Блок установки: шаги для Telegram, фильтрация приложений, кнопки «Add proxy» / копирование proxy-ссылки, QR (`panel/components/sub/blocks/InstallationGuideBlock.tsx`, `AddToAppBlock.tsx`).
  - Настройки страницы: флаг **`mtProtoEnabled`** в response rules (редактор + схема Zod + поле в `SharxSubpageResponseRules` на backend, `web/service/sharx_subpage_config.go`).

- **Ресурсы хоста на дашборде (панель + узлы)**
  - Новый endpoint **`GET /panel/api/server/memHistory/{bucket}`** — история использования **памяти хоста в процентах** (`web/controller/server.go`).
  - **`GET /panel/api/server/cpuHistory/{bucket}`** и **memHistory** в multi-node возвращают структуру **`{ series: [...] }`**: серия для панели и по одной на worker-узел с `key`, `name`, `nodeId`, `points` (поле времени **`t`**, для CPU — `cpu`, для памяти — `mem`).
  - Сервис: кольцевые буферы для памяти панели, агрегаты и поллинг статусов worker’ов (`web/service/server.go`); фоновый опрос `pollWorkerHostMetricsIfDue` с rate-limit.
  - Дашборд: многосерийные спарклайны и графики, обратная совместимость со старым форматом «массив точек» (`panel/components/DashboardPage.tsx`).

- **Node: host CPU / memory в `/api/v1/status`**
  - Новый модуль **`node/api/hoststatus.go`**: `hostCpu`, `hostMem` (`current` / `total`) мержатся в JSON статуса (`node/api/server.go`).

### Improvements

- **Multi-node / инбаунды:** при проверке «тот же порт на узле» учитывается ещё и **`listen`**, чтобы разные listen на одном порту не считались конфликтом (`web/service/node.go`).
- **Публичная подписка / редактор:** i18n placeholder’ов и переключатель MTProto в response rules (`ResponseRulesEditor.tsx`, смежные редакторы подписки).
- **API documentation:** описаны новый формат **cpuHistory** и endpoint **memHistory** (`web/docs/API.md`).
- **Локализация:** обновлены `panel/public/locales/*.json` и `web/translation/*.toml` (в т.ч. крупное обновление `translate.en_US.toml` / `translate.ru_RU.toml`).

### Fixes / API behavior

- Ответы HMAC API при неизвестном/выключенном узле и ошибках сборки конфига используют **I18nWeb** вместо жёстко заданных английских строк (`web/controller/api.go`).

### Breaking / Operational Notes

- **Клиенты `cpuHistory`:** ожидаемый контракт ответа изменён с «массива точек» на объект **`{ series: [...] }`**. Фронт панели в этом релизе поддерживает оба варианта; внешние интеграции нужно обновить.
- **Поле времени в точках:** документация и новый формат используют **`t`** (unix seconds); старые клиенты, ожидающие только `time`, могут потребовать доработки (дашборд читает `t` или `time`).

### 📦 Upgrade from v1.3.2

1. Сделайте **бэкап** базы данных и `.env`.
2. Поднимите панель и node на версии **`1.3.3`** (образы **`1.3.3`** / **`latest`**).
3. Миграции не требуются.
4. Проверьте:
   - публичную HTML-подписку: наличие **`mtProtoLinks`** и блок Telegram при включённом MTProto в response rules;
   - дашборд: CPU и память, в multi-node — отдельные серии по узлам;
   - статус node: поля **`hostCpu`** / **`hostMem`**;
   - при необходимости обновите потребителей **`cpuHistory`** под формат **`series`**.

```yaml
services:
  sharx:
    image: {{SHARX_TAG_LATEST}}
  sharxnode:
    image: {{SHARXNODE_TAG_LATEST}}
  postgres:
    image: {{HARBOR_HOST}}/{{HARBOR_PROJECT}}/postgres:latest
```
