## Release v1.4.4

**Release Date:** May 28, 2026

Релиз **v1.4.4** относительно **v1.4.3**: патч-релиз с исправлением сохранения IP-лимита у клиента — галочка `Limit concurrent IPs` и поле `Max unique IPs` не персистились в БД при апдейте клиента и каждый раз возвращались к дефолтам после переоткрытия карточки. На стороне xray и telemt enforcement продолжал использовать старое значение (`ip_limit_enabled=false`), поэтому лимит фактически не применялся даже после явного включения в UI.

Миграций БД **нет**.

---

### Bug Fixes

- **`ipLimitEnabled` / `maxIPs` не сохраняются при апдейте клиента — лимит IP не работает ни для xray, ни для telemt** (`web/service/client.go`)
  - **Причина:** `ClientService.UpdateClient` строит `updates map[string]interface{}` для `gorm.Updates(...)` и явно перечисляет столбцы, которые нужно обновить (`enable`, `total_gb`, `tg_id`, `hwid_enabled`, `max_hwid`, ...). Колонки `ip_limit_enabled` и `max_ips` (добавлены миграцией `0043` в v1.3.9) **не были включены в эту мапу** — поэтому GORM их молча игнорировал, а в БД оставались дефолты (`false` / `1`). Все следующие чтения (`/panel/client/get/{id}`) возвращали `ipLimitEnabled: false, maxIPs: 1`, и в UI карточки галочка «отскакивала».
  - **Следствие на enforcement:**
    - **Telemt MTProto** (`web/service/telemt_config.go`): `BuildTelemtToml` пишет `u.MaxUniqueIPs = c.MaxIPs` только если `c.IPLimitEnabled && c.MaxIPs > 0`. С `IPLimitEnabled=false` в БД генерируемый TOML на ноде не содержал лимит и сайдкар (`ip_tracker.rs`) не блокировал лишние IP.
    - **Xray (vless/trojan/ss/…)** (`web/job/check_client_ip_limit_job.go`): job отбирает клиентов по `WHERE ip_limit_enabled = true AND max_ips > 0 AND enable = true`. С `IPLimitEnabled=false` в БД клиент в выборку не попадал, и панель не слала ноде `/api/v1/drop-ips` для лишних IP.
  - **Фикс:** `updates["ip_limit_enabled"] = client.IPLimitEnabled` и `updates["max_ips"] = client.MaxIPs` добавлены в всегда-обновляемые поля (паттерн `enable` / `total_gb` — `false`/`0` это валидные пользовательские значения, GORM `Updates(struct)` иначе их выкидывает как zero-value). Fallback на отсутствующие столбцы расширен — если БД пропустила миграцию `0040` (HWID) или `0043` (IP limit), эти колонки удаляются из апдейта и операция повторяется (как раньше с `hwid_enabled` / `max_hwid`).

### Improvements

- **Документация механики IP-лимита** (в этом release notes, ниже)
  - **Telemt** enforce'ит лимит **локально на ноде**: `u.MaxUniqueIPs` в TOML → `IpTracker` (`third_party/telemt-sharx/src/ip_tracker.rs`) держит reservoir per-user и при превышении сбрасывает новые соединения.
  - **Xray** не имеет нативного per-user IP-лимита; enforcement делается **в две стороны**: периодический `CheckClientIPLimitJob` на панели опрашивает сессии клиента со всех нод, при `len(ips) > maxIPs` шлёт ноде `DropSessionsByIPsForClient(excess)` → нода через Xray API физически закрывает соединения с лишних IP. То есть _координирует_ панель, _дропает_ нода.

### Operational Notes

- Обновление затрагивает только **панель** (`sharx`) — нода (`sharxnode`) обновления не требует.
- Существующие клиенты с пустым/потерянным IP-лимитом — после обновления откройте карточку клиента, выставьте галочку `Limit concurrent IPs` и нужный `Max unique IPs`, сохраните. Запись закрепится в БД, и на следующем apply-config telemt получит TOML с `max_unique_ips`, а xray-job начнёт включать клиента в выборку.
- Telemt-инбаунд требует **apply-config** (или автоматический pull) для подхвата нового TOML — рестарт сайдкара происходит автоматически по diff'у.

### 📦 Upgrade from v1.4.3

1. Бэкап `.env` (миграции БД не требуются).
2. Обновите образ **`sharx`** до версии **`1.4.4`** / **`latest`**. Образ **`sharxnode`** обновления не требует.
3. Проверьте:
   - В карточке клиента включите `Limit concurrent IPs`, задайте `Max unique IPs`, сохраните, переоткройте карточку — галочка остаётся включённой, число сохранено.
   - В БД: `SELECT id, name, ip_limit_enabled, max_ips FROM client_entities WHERE id = <client_id>;` отражает выставленные значения.
   - Для telemt-инбаунда: после apply-config TOML на ноде содержит `max_unique_ips = <N>` под нужным `[[users]]`.
   - Для xray-инбаундов: логи панели `CheckClientIPLimitJob: client <name> exceeded IP limit (X>N), dropping K IP(s)` появляются при превышении.

services:

    image: {{SHARXNODE_TAG_LATEST}}
  postgres:
    image: {{HARBOR_HOST}}/{{HARBOR_PROJECT}}/postgres:latest
