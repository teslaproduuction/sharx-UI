## Release v1.4.5

**Release Date:** May 28, 2026

Релиз **v1.4.5** относительно **v1.4.4**: hot-fix допилки IP-лимита из v1.4.4. В v1.4.4 был исправлен слой сервиса (`ClientService.UpdateClient` — теперь персистит `ip_limit_enabled` / `max_ips` в БД), но **контроллер** `web/controller/client.go::updateClient` так и не передавал эти поля в сервис: JSON-путь вручную копирует whitelisted-поля из request body в `client *model.ClientEntity`, и в этом списке отсутствовали `ipLimitEnabled` / `maxIPs`. Поэтому сервис получал значение `existing` (`false` / `1`) и именно его и сохранял — галочка по-прежнему «отскакивала» после reopen карточки.

Миграций БД **нет**.

---

### Bug Fixes

- **`ipLimitEnabled` / `maxIPs` не доходят до сервиса при апдейте клиента — лимит IP всё ещё не работает** (`web/controller/client.go`)
  - **Причина:** `updateClient` хэндлер для JSON-запросов парсит `updateData map[string]interface{}` и поэлементно копирует поля на `client *model.ClientEntity` (паттерн «только то, что прислал клиент, остальное берём из existing»). В списке копируемых полей были `hwidEnabled` / `maxHwid` (HWID), но **не было `ipLimitEnabled` / `maxIPs`**. Аналогично — form-путь не имел соответствующих `c.PostForm("ipLimitEnabled")` / `c.PostForm("maxIPs")`. В результате до `ClientService.UpdateClient` доезжали значения `existing.IPLimitEnabled = false`, `existing.MaxIPs = 1`, и фикс v1.4.4 на уровне сервиса корректно сохранял именно их.
  - **Следствие:** наблюдаемое поведение полностью идентично v1.4.3 — сразу после `POST /panel/client/update/{id}` ответ `GET /panel/client/get/{id}` возвращал `ipLimitEnabled: false, maxIPs: 1` независимо от того, что прислал UI; telemt-TOML не содержал `max_unique_ips`, xray-job не выбирал клиента в `WHERE ip_limit_enabled = true`.
  - **Фикс:** в JSON-ветке `updateClient` добавлен парсинг `ipLimitEnabled` (как bool) и `maxIPs` (float64/int/int64, чтобы переварить любой JSON-numeric тип, паттерн идентичен `maxHwid`). В form-ветке добавлены `c.PostForm("ipLimitEnabled")` и `c.PostForm("maxIPs")` с теми же правилами «`""` = не передано → keep existing». Сервисный фикс из v1.4.4 теперь действительно срабатывает.

### Operational Notes

- Обновление затрагивает только **панель** (`sharx`) — нода (`sharxnode`) обновления не требует.
- Если перед обновлением вы пытались сохранить IP-лимит в v1.4.3 / v1.4.4 и он не сохранялся — после раскатки v1.4.5 откройте карточку клиента ещё раз, проставьте `Limit concurrent IPs` + `Max unique IPs`, сохраните. Теперь значения корректно дойдут до БД.
- Самопроверка: после save сделайте `GET /panel/client/get/{id}` (или просто переоткройте карточку) — должно вернуть `ipLimitEnabled: true, maxIPs: <N>`. В БД `SELECT id, ip_limit_enabled, max_ips FROM client_entities WHERE id = <id>;` тоже отражает значения.
- Для telemt-инбаунда: на ближайшем `apply-config` TOML на ноде получит `max_unique_ips = <N>`; сайдкар рестартует автоматически по diff'у. Для xray-инбаундов: `CheckClientIPLimitJob` начнёт включать клиента в выборку и шлёт ноде `drop-ips` при превышении.

### 📦 Upgrade from v1.4.4

1. Бэкап `.env` (миграции БД не требуются).
2. Обновите образ **`sharx`** до версии **`1.4.5`** / **`latest`**. Образ **`sharxnode`** обновления не требует.
3. Проверьте:
   - В карточке клиента включите `Limit concurrent IPs`, задайте `Max unique IPs`, сохраните, переоткройте карточку — галочка остаётся включённой, число сохранено.
   - В БД: `SELECT id, name, ip_limit_enabled, max_ips FROM client_entities WHERE id = <id>;` отражает выставленные значения.
   - Для telemt-инбаунда: после apply-config TOML на ноде содержит `max_unique_ips = <N>` под нужным `[[users]]`.
   - Для xray-инбаундов: при превышении лимита в логах панели появляется `CheckClientIPLimitJob: client <name> exceeded IP limit (X>N), dropping K IP(s)`.
