## Release v1.3.9

**Release Date:** May 28, 2026

Релиз **v1.3.9** относительно **v1.3.8**: крупный релиз с несколькими **критически важными изменениями базы данных**, новыми функциями и улучшениями UI.

> ### ⚠️ ОБЯЗАТЕЛЬНО: Снимите дамп базы данных перед обновлением
>
> В этом релизе присутствуют **две миграции базы данных**:
> - `0042` — переименование колонки `email` → `name` в таблице `client_entities` и `client_email` → `client_name` в `inbound_client_ips`. Это **необратимое изменение схемы**.
> - `0043` — добавление колонок `ip_limit_enabled` и `max_ips` в таблицу `client_entities`.
>
> **Перед обновлением обязательно выполните:**
> ```bash
> docker exec -t sharx-postgres pg_dump -U sharx sharx > backup_$(date +%Y%m%d_%H%M%S).sql
> ```
> Или через стандартный бэкап `.env` + БД. Убедитесь, что бэкап прошёл успешно, прежде чем запускать новые образы.

### Breaking Changes

- **Переименование поля `email` → `name` у клиентов** (`database/model/model.go`, `database/migrations/0042_client_name_rename.sql`)
  - В модели `ClientEntity` поле `Email` переименовано в `Name` (JSON: `"name"`), что точнее отражает его роль — уникальный неизменяемый идентификатор клиента в рамках пользователя панели.
  - В таблице `client_entities`: `ALTER TABLE ... RENAME COLUMN email TO name`, индекс `idx_user_email` → `idx_user_name`.
  - В таблице `inbound_client_ips`: колонка `client_email` → `client_name`.
  - Изменение распространяется на все сервисы: `web/service/client.go`, `web/service/client_session.go`, `web/service/client_session_block.go`, `sub/subService.go`, `sub/subJsonService.go`, `sub/wireguard_panel.go`, `web/job/check_client_ip_job.go`, `web/controller/client.go` и другие.
  - UI (`panel/components/ClientsPage.tsx`): колонка и фильтр переименованы из «Email» → «Client name».

### New Features

- **Лимит одновременных уникальных IP на клиента** (`web/job/check_client_ip_limit_job.go`, `database/migrations/0043_client_ip_limit.sql`)
  - Новый фоновый джоб `CheckClientIPLimitJob` отслеживает количество уникальных IP-адресов в активных сессиях каждого клиента.
  - Если клиент превышает заданный лимит (`max_ips`), лишние сессии принудительно сбрасываются.
  - Функция независима от HWID-лимита и управляется отдельным флагом `ip_limit_enabled` / полем `max_ips` на клиенте.
  - Новые колонки в `client_entities`: `ip_limit_enabled BOOLEAN DEFAULT false`, `max_ips INTEGER DEFAULT 1`.

- **Управляемая ротация лог-файлов** (`logger/rotate.go`, `web/service/log_rotate.go`, `node/api/log_rotate.go`)
  - Добавлена конфигурируемая ротация лог-файлов панели и нод на базе `lumberjack`.
  - Параметры задаются в настройках панели (Settings → Logging):
    - **Log max size (MB)** — ротация при превышении размера (1–1024 МБ, дефолт 50).
    - **Log retention (days)** — удаление старых файлов через N дней (1–365, дефолт 14).
    - **Rotated log files** — количество хранимых ротированных файлов (1–100, дефолт 5).
    - **Compress rotated logs** — сжатие (gzip) старых файлов для экономии места.
  - При сохранении настроек конфигурация ротации автоматически применяется на панели и передаётся на ноды через `apply-config`-конверт.

- **Ресурсный монитор ноды** (`panel/components/NodeResourceDrawer.tsx`, `panel/components/NodesPage.tsx`)
  - На странице Nodes добавлена кнопка (иконка Activity) для каждой ноды — открывает drawer с графиками CPU / RAM / Disk в реальном времени.
  - Dashboard теперь отображает метрики только панели (`panelOnlySeries`), а не агрегат со всеми нодами — для раздельного наблюдения.

### Improvements

- **Hysteria2: поддержка того же порта** (`web/service/node.go`) — [@silverfox0000]
  - Снято ограничение: Hysteria2 теперь может слушать на том же порту, что и другие inbound'ы ноды.

- **Поля Encryption / Decryption в UI инбаундов** (`panel/components/InboundsPage.tsx`, `panel/lib/inboundDefaults.ts`) — [@silverfox0000]
  - В форме создания/редактирования инбаунда добавлены поля `encryption` и `decryption` для соответствующих протоколов.
  - Закрывает issue [#81 — «Обфускация hysteria2»](https://github.com/konstpic/SharX/issues/81): поля обфускации (obfs: salamander) для Hysteria2 теперь доступны прямо в форме инбаунда.

- **Расширенный список типов заголовков KCP и QUIC** (`panel/components/InboundsPage.tsx`)
  - Отдельный список `KCP_HEADER_TYPES` для mKCP.
  - QUIC header types дополнены: `srtp (VoIP)`, `utp (BitTorrent)`, `wechat-video`, `dtls`, `wireguard` — с понятными метками.

- **Редактируемые HTTP-заголовки TCP stream** (`panel/components/InboundsPage.tsx`)
  - В настройках TCP-stream можно добавлять/удалять произвольные HTTP-заголовки запроса (`tcpHttpRequestHeaders`) и ответа (`tcpHttpResponseHeaders`) прямо из UI.

- **`acceptProxyProtocol` в конфигурации инбаунда** (`xray/inbound.go`)
  - Поле `acceptProxyProtocol` добавлено в `InboundConfig` — позволяет принимать соединения с PROXY protocol (HAProxy и аналоги).

- **Исправление мигратора: только новые миграции** (`database/migrator.go`)
  - Мигратор переработан: теперь пропускает уже применённые версии (`applied[migration.Version]`) вместо повторного запуска всех миграций. Устранён шум в логах о «re-applied» миграциях.

- **WireGuard: расширенный поиск peer по имени** (`sub/wireguard_panel.go`)
  - Функция поиска peer принимает несколько ключей: `name`, `clientName`, `panelName`, `email`, `clientEmail`, `panelEmail` — для совместимости с конфигурациями, созданными до переименования.

### Resolved Issues

- [**#81** — Обфускация hysteria2](https://github.com/konstpic/SharX/issues/81) — добавлены поля `encryption` / `decryption` в форму инбаунда, в том числе `obfs: salamander` для Hysteria2.

### Operational Notes

- Обновление **требует** обновления **и панели, и worker-образов** до **1.3.9**: миграции применяются автоматически при старте.
- Ротация логов работает «из коробки» с дефолтными значениями; тонкую настройку можно изменить через Settings.
- IP-лимит по умолчанию **отключён** (`ip_limit_enabled = false`) для всех существующих клиентов — включайте вручную по необходимости.
- После обновления рекомендуется проверить страницу Clients: колонка «Email» теперь называется «Client name».

### 📦 Upgrade from v1.3.8

> **Шаг 0 (обязательно):** снимите дамп БД.
> ```bash
> docker exec -t sharx-postgres pg_dump -U sharx sharx > backup_v1.3.8_$(date +%Y%m%d_%H%M%S).sql

1. Бэкап БД и `.env` (см. выше — это обязательный первый шаг).

   - Страница Clients: колонка «Client name» отображается корректно.
